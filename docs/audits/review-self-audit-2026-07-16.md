## 1. Claims fully retracted

- **"Crypto sync still writes only legacy `Holding`."** False at the time of writing. The BTC sync dual-writes: legacy `Holding` upsert *and* canonical `PositionObservation` (instrument-resolved, idempotent, flag-gated, flag set). I took this from STATUS.md's Phase-0 block instead of reading `btc-sync.ts`, which had already been updated by commit `ee38377`. I was wrong.
- **"Activation impossible by configuration" / dark flags in no env file** (repeated from the 07-12 repository audit). The three A-series flags are documented in `.env.example` and set in `.env.local`. The doc-era claim was stale when I repeated it.
- **"A9 has no trigger wiring."** `regenerateWealthHistoryForAccounts` is called from `jobs/sync-banks.ts:114`, `jobs/sync-crypto.ts`, and `lib/plaid/backgroundHistorySync.ts:199`. My report hedged this in one section and asserted it in the technical-debt list; the assertion is withdrawn.

- **"InvestmentsSpaceData has zero consumers"** → zero consumers *of the composed envelope*; the contract's historical binding (`loadInvestmentsHistory`, same file, same contract) is the live production path for the Time Machine route. The premature-abstraction charge is downgraded to an activation-gap charge: the contract is real, composed, and test-pinned; only the UI cutover (doctrine Phase 5) is missing.
- **"Two loaders and a memo" (Connections)** → Connections is a wired end-to-end contract (server page load, sanctioned poll route, props-only children, connection-id keying, deliberate user-scoping). What survives: the two contracts still share no base type, and the shared naming convention is still enforced by comments citing other files rather than by an import.
- **URL-writer attribution**: the `?transaction` writer is `useTransactionDrawer` (via `router.push`/`replace`), not `DashboardChrome` itself. The ≥4-independent-writers finding stands (5 raw history calls in the host, 4 in the shell hook, plus the drawer).

## 4. New risks discovered during the re-check

1. **Crypto now has three concurrent valuation sources.** Wealth/snapshots read `FinancialAccount.balance` (per `btc-sync.ts:183`, "what Wealth still reads this slice"); the canonical spine holds `PositionObservation` quantities; AI/export consume a *union* of `getCurrentPositions` plus the `readLegacyCryptoWalletPositions` bridge with in-code dedup (`holdings.ts:19-30`). The dual-write fixed the spine; it also created a three-way read split that exists until the Part-9 census gate closes — and that gate is not close: `getHoldings` (`lib/data/accounts.ts:241`) still reads `db.holding` and Wealth still reads balances.
2. **The Investments workspace's "current" state is presence-counts only.** `getInvestmentAccountsView` deliberately reads position *counts*, not contents. So the product currently ships a Time-Machine-derived current view plus a count-based account list, while the true canonical current portfolio (`assembleCurrentPortfolio`) is invisible. The activation gap is wider than "cutover pending" — there is no user-facing surface for the system's best current-position data.
3. **The intentional bucket-taxonomy split is now test-pinned.** `valuation.investment-bucket.test.ts:57` *asserts* that A10 does not exclude digital assets while the snapshot path does. A future engineer "fixing" the investments/wealth discrepancy will find a green test defending each side. Divergence has been promoted from accident to doctrine, which makes it harder, not easier, to unwind.
4. **My own review is now part of the doctrine-drift problem.** It repeated doc-era claims the code had already fixed, and it sits in the repo root beside the audits it criticized. The project's documented failure mode — prose diverging from code — now includes an external reviewer's document. The mitigation is the same one I prescribed for the project: claims about code must cite code.
