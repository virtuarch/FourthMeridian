# Hydration Rules

React hydration fails when the HTML rendered on the server does not match the first render on the client. In Fourth Meridian, the most common causes are:

1. **Timezone-sensitive date formatting** ‚Äî `new Date(iso).toLocaleDateString()` with no explicit timezone produces different calendar dates on a Mac dev server (e.g. UTC-5) vs an Android phone (e.g. UTC+5:30) for timestamps near midnight UTC.
2. **ICU version drift in combined date+time strings** ‚Äî `Intl.DateTimeFormat.format()` with both date and time fields produces different separators across Node/ICU versions ("Jun 9, 2026, 9:00 AM" on ICU 71 vs "Jun 9, 2026 at 9:00 AM" on ICU 73+). Use `formatToParts` + manual assembly instead.
3. **Local-hour greetings** ‚Äî `new Date().getHours()` returns a different value on the server than on the phone.
4. **Browser/password-manager attribute injection** ‚Äî Chrome injects `__gcruniqueid` into recognized autofill inputs before React hydrates.

## Date Formatting

### Approved pattern ‚Äî UTC-stable helpers from `lib/format.ts`

```tsx
import { formatDate, formatMonthYear, formatDateTime } from "@/lib/format";

// "Jan 15, 2024"
<p>{formatDate(account.lastUpdated)}</p>

// "January 2026"
<p>{formatMonthYear(goal.targetDate)}</p>

// "Jan 15, 2024 at 10:30 AM"
<p>{formatDateTime(auditEntry.createdAt)}</p>
```

All helpers in `lib/format.ts` pass `timeZone: "UTC"` to `Intl.DateTimeFormat`, so server and client always produce the same string for the same ISO timestamp.

**Date+time helpers (`formatDateTime`, `formatDateTimeShort`) use `formatToParts` + manual assembly** to avoid ICU separator drift. Do not call `.format()` directly on a formatter that includes both date and time fields.

### Forbidden pattern ‚Äî bare `toLocaleDateString()`

```tsx
// ‚ùå NEVER ‚Äî produces different strings in different timezones
<p>{new Date(iso).toLocaleDateString("en-US", { month: "short", ... })}</p>
```

### Forbidden pattern ‚Äî `.format()` with combined date+time fields

```tsx
// ‚ùå NEVER ‚Äî separator between date and time varies across Node/ICU versions
//   Node 18 / ICU 71 ‚Üí "Jan 15, 2024, 10:30 AM"
//   Node 22 / ICU 73 ‚Üí "Jan 15, 2024 at 10:30 AM"
new Intl.DateTimeFormat("en-US", { ..., hour: "numeric", minute: "2-digit" }).format(d)

// ‚úì Use formatDateTime() from lib/format.ts ‚Äî uses formatToParts + manual assembly
import { formatDateTime } from "@/lib/format";
<p>{formatDateTime(entry.createdAt)}</p>
```

### Forbidden pattern ‚Äî `toLocaleString()` without explicit locale

```tsx
// ‚ùå NEVER ‚Äî locale falls back to the runtime's locale, which differs
<p>{value.toLocaleString()}</p>

// ‚úì Use instead:
import { formatNumber } from "@/lib/format";
<p>{formatNumber(value)}</p>
```

## Greeting (time-of-day)

The greeting ("Good morning", "Good afternoon", "Good evening") depends on `new Date().getHours()`, which differs between the server and the user's device.

### Approved pattern ‚Äî compute after mount

```tsx
import { getGreeting, GREETING_PLACEHOLDER } from "@/lib/format";

const [greeting, setGreeting] = useState(GREETING_PLACEHOLDER); // "Welcome back" on SSR
useEffect(() => { setGreeting(getGreeting()); }, []);            // replaced after mount

// In JSX ‚Äî no suppressHydrationWarning needed:
<h1>{greeting}, {firstName}</h1>
```

`GREETING_PLACEHOLDER = "Welcome back"` is rendered both on the server and on the first client paint, so there is no mismatch. The actual greeting swaps in silently after hydration.

### Forbidden pattern ‚Äî call during render

```tsx
// ‚ùå NEVER ‚Äî new Date().getHours() differs on server vs phone
<h1 suppressHydrationWarning>{getGreeting()}, {firstName}</h1>
```

## Root `<html>` Element

Chrome mobile / Google remote-frame tooling injects attributes such as `__gcrremoteframetoken` directly onto the root `<html>` element before React hydrates. This is not generated by app code.

`app/layout.tsx` carries `suppressHydrationWarning` on `<html>` for exactly this reason:

```tsx
<html lang="en" className="h-full" suppressHydrationWarning>
```

This suppression applies only to attributes on the `<html>` tag itself. React's `suppressHydrationWarning` is shallow ‚Äî it does not propagate to child elements ‚Äî so no app-level mismatches elsewhere are hidden by it.

## Password Manager / Autofill Inputs

Chrome injects `__gcruniqueid` into recognized `<input>` elements **and their parent `<form>`** before React hydrates. Both the `<form>` and the `<input>` nodes need `suppressHydrationWarning`.

### Approved

```tsx
// Both form and inputs need suppression ‚Äî Chrome targets both
<form onSubmit={handleSubmit} suppressHydrationWarning>
  <input
    type="text"
    autoComplete="username"
    suppressHydrationWarning
    ...
  />
</form>
```

Apply to every `<form>` that contains autofill-recognized fields (email, username, password, one-time-code).

---

## `suppressHydrationWarning` ‚Äî Allowed vs Forbidden

| Context | Allowed? | Why |
|---|---|---|
| `<html>` in `app/layout.tsx` | ‚úÖ | Chrome mobile injects `__gcrremoteframetoken` before hydration |
| `<input>` with `autoComplete` | ‚úÖ | Browser/password-manager injects `__gcruniqueid` before hydration |
| `<form>` containing autofill inputs | ‚úÖ | Chrome also injects `__gcruniqueid` on the `<form>` element itself |
| `<p>` / `<span>` rendering UTC-stable formatted dates | ‚ùå | Use `formatDate()` ‚Äî no mismatch possible |
| `<h1>` rendering greeting | ‚ùå | Use `useState(GREETING_PLACEHOLDER)` + `useEffect` |
| `<p>` rendering `payoffDate` from `new Date()` | ‚úÖ | Projection depends on current moment ‚Äî inherently local |
| Large container `<div>` | ‚ùå | Hides unknown structural bugs |
| `className` mismatch | ‚ùå | Fix the root cause |

**Rule of thumb:** `suppressHydrationWarning` is acceptable only on exact leaf nodes where the mismatch is *known*, *intentional*, and *unavoidable* ‚Äî and the value is either injected by a third party (browser extension, password manager) or genuinely depends on the client's local time after mount.

## Adding New Date Display

1. Import from `@/lib/format`:
   ```tsx
   import { formatDate } from "@/lib/format";
   ```
2. Use it directly in JSX ‚Äî no `suppressHydrationWarning` needed:
   ```tsx
   <p>Updated {formatDate(account.lastUpdated)}</p>
   ```
3. If the format you need doesn't exist in `lib/format.ts`, add it there with `timeZone: "UTC"` ‚Äî do not inline a new `toLocaleDateString` call.

## Adding New Number Display

```tsx
import { formatNumber, formatCurrency } from "@/lib/format";

<p>{formatNumber(userCount)}</p>          // "1,234"
<p>{formatCurrency(accountBalance)}</p>   // "$1,234"
```

Never use bare `.toLocaleString()` without an explicit locale argument.
