## 3. The five questions

**1. Biggest architectural accomplishment of v2.5.**
Closing the multi-tenant account-sharing seam end to end: the `WorkspaceAccountShare → SpaceAccountLink` migration was completed *and the legacy model retired* (`e17c699`), and every read surface (AI assemblers, UI data layer, account/holdings reads) was unified behind one canonical visibility predicate (`lib/ai/visibility.ts`), closing all four leak classes (KD-1/15/19) with a green two-user proof. That is the expensive, correctness-critical work, and it landed. (Design-track runner-up: unifying every overlay onto `OverlaySurface`.)

**2. Biggest remaining blocker before v2.5 can be called complete.**
The v2.4.5 verification gate was never formally closed, yet v2.5 opened anyway — and the design-foundation work is now a large **uncommitted, partly-experimental working tree** with a STATUS.md that no longer matches the branch. Concretely: the named test suites (merchant/rollup/heuristic math) and observability counters don't exist, the Liquid experiment concluded but wasn't cleaned up, and the two shipped initiatives (Material Engine, Perspective Engine) aren't reconciled into the canonical doc. There is no single trustworthy statement of current state — which blocks any confident "complete" call.

**3. What must NOT be started until that blocker is finished.**
Anything that builds new product surface or acts unprompted on top of unverified foundations: v2.6b ambient intelligence (scheduler wiring, `AiAdvice` write path, notifications), AI-5 advisor conversation layer, and the D2.x background sync pipeline. Also do **not** retire legacy `Account`, and do not begin MC1 multi-currency conversion. Adding scheduled/autonomous behavior before the verification gate and the state-of-truth are settled compounds risk.

**4. What is safe to begin in parallel.**
Additive, non-behavioral work: writing the missing v2.4.5 test suites; adding observability counters; the max-50 copy fix and `.env.example` flag documentation; committing/pruning the Liquid dead-end and its npm deps; DataCard Step B consumer migration and finishing material-engine adoption (additive, per-surface, revertible); reconciling STATUS.md (log the Material Engine + Perspective Engine, fix the version/phase fields); removing the `… 2` duplicate dirs.

**5. Implementation order if taking over today.**
(1) **Reconcile reality** — commit or discard the working tree, prune dead-end experiments/deps, remove `… 2` dirs, update STATUS.md §2/§3 to match the branch (log Material Engine + Perspective Engine). (2) **Close v2.4.5 for real** — the three named test suites, observability counters, max-50 copy, flag docs; then tag it. (3) **Finish v2.5 design foundation** — DataCard Step B, material-engine adoption, settle the single Brief card/CTA architecture, Phase 1B if wanted; ship new surfaces in the language. (4) **v2.5.5 Financial Intelligence** — flowType + semantics doctrine + KD-6. (5) **D2.x Initial Sync Experience.** (6) **v2.6a AI-5**, then (7) **v2.6b ambient** (scheduler/advice), (8) **MC1**, (9) **v3.0 launch**.
