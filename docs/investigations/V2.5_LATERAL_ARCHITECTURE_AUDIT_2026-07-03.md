# Fourth Meridian v2.5 — Lateral Architecture Audit

**Type:** Investigation only — no implementation, no schema edits, no migrations, no code.
**Date:** 2026-07-03
**Baseline:** `feature/v2.5-spaces-completion` @ `f4ab6a9`; STATUS.md as canonical state authority.
**Purpose:** Identify architectural work that can proceed **in parallel** with the active FlowType lane, confined to its own subsystem, with low merge-conflict risk and long-term leverage.

## Active lanes explicitly excluded from every recommendation below

Per the audit constraints, nothing here touches: **FlowType** (v2.5.5, active), **Merchant normalization** (ongoing investigation), **Atlas Liquid / UI-1** (Phase 1 committed), **Daily Brief**, or **Space Dashboard UI**. Where an area's most obvious next step lands in one of those lanes (e.g. the Cash Flow perspective, transaction-semantics fixes KD-17/18), it is named as blocked and skipped, not recommended.

## Method

Grounded in a full read of `prisma/schema.prisma` (1,479 lines, all models/enums), the D4 AI context stack (`lib/ai/*`), the Perspective Engine (`lib/perspective-engine/*`), provider abstraction (`lib/providers/*`, `lib/plaid/*`, `lib/accounts/*`), encryption (`lib/plaid/encryption.ts`), snapshots (`lib/snapshots/regenerate.ts`, `jobs/*`), Space infrastructure (`lib/space*.ts`, `lib/widget-registry.ts`), and the governing decision records (`docs/architecture/PHASE_2_*`, STATUS.md §3 ledger and §7 defect register). Every claim cites a file or a ledger row.

# Part A — Area Audits

## 1. Perspective Engine

**State (grounded).** PE1 is *functionally complete* (STATUS §3): a deterministic, non-persistent lens layer (`lib/perspective-engine/`) with guard-tested invariants — byte-identical output under an injected clock, no direct Prisma import, no LLM math, reads exclusively through the KD-19 visibility-enforced data layer (`lib/data/accounts.ts#getAccountsWithVisibility`), and name-free-by-construction provenance. Two lenses ship: **Liquidity** and **Debt** (`lenses/{liquidity,debt}.core.ts`). The registry (`registry.ts`) mirrors the assembler registry deliberately, so the pattern is already proven twice.

**Can PE become more useful before FlowType lands? Yes — deterministically, right now.** The feasibility matrix (`PERSPECTIVE_ENGINE_FOUNDATION_INVESTIGATION.md` §3) already classifies the candidate set. Two lenses are safe today and touch nothing FlowType owns:

- **Net Worth (attribution/trend)** — "Safe now," reads `SpaceSnapshot` (pre-aggregated). The investigation deferred it from slice 1 only because the attribution *copy* wasn't designed, not because of any data or conflict blocker.
- **Investments (exposure/concentration)** — "Safe later": the deterministic math (concentration, Herfindahl, FULL-only positions) already exists in the holdings assembler (`lib/ai/assemblers/holdings.ts`); the only caution is Holding's mid-D11 dual-anchor state (see Bottleneck #1), which argues for extracting a shared holdings-math module first rather than duplicating.

**Cash Flow is correctly blocked** by v2.5.5 flowType (settlement/flow semantics) and is out of scope for this audit.

### Recommendation PE-2 — Net Worth (attribution) lens

- **Purpose:** Answer "what is my net worth, and what's driving its change?" as a deterministic, provenance-carrying verdict, reusing the existing `SpaceSnapshot` time series.
- **Why now:** The engine, registry, contract (assumptions/provenance/empty-state), and test harness already exist; adding a lens is the cheapest high-leverage AI-adjacent work available that requires zero new data model. It also exercises the snapshot read path, surfacing the Snapshot gaps in Area 6 before they bite a shipped feature.
- **Dependency graph:** Depends only on `SpaceSnapshot` (exists) and the PE registry (exists). No dependents block on it. Loosely *enabled by* Area 6 snapshot-coverage work (better history → better verdict) but not blocked by it.
- **Merge-conflict risk:** **Very low.** New files under `lib/perspective-engine/lenses/` + one `lensId` attachment in `lib/perspectives.ts` + one route import. The README documents this exact additive procedure. No shared file is rewritten.
- **Architectural payoff:** Proves the engine generalizes past account-balance lenses to time-series lenses; establishes the attribution-copy contract that future lenses inherit; gives Daily Brief / D4 a deterministic net-worth verdict to consume later without recomputation.
- **Estimated size:** **S** (~1 core file + 1 binding + 1 test suite; the matrix says the math is "deterministic trend today").
- **Recommended order:** First PE lane item; ship before the Investments lens.

### Recommendation PE-3 — Extract shared holdings-math module, then Investments lens

- **Purpose:** Lift concentration/Herfindahl/exposure math out of the holdings assembler into a pure shared module, then bind an Investments perspective to it.
- **Why now:** Duplicating assembler math into a lens "invites drift" (investigation §3). Extraction is pure-refactor leverage that also de-risks the D11 Holding dual-anchor retirement (Bottleneck #1) by centralizing the read.
- **Dependency graph:** Extraction depends on nothing; the lens depends on the extraction. Best done *after* the Holding legacy anchor retires, or paired with a read-normalization guarantee.
- **Merge-conflict risk:** **Low-medium** — touches `lib/ai/assemblers/holdings.ts` (shared with the D4 lane). Coordinate so it doesn't collide with any concurrent assembler edit; otherwise isolated.
- **Architectural payoff:** One canonical holdings-math implementation feeding both the AI context and the perspective layer; removes a drift class.
- **Estimated size:** **M** (extraction + lens + tests + assembler rewire).
- **Recommended order:** After PE-2 and after (or jointly with) the Holding dual-anchor cleanup.

## 2. Provider abstraction (D2)

**State (grounded).** D2 is *Active — production hardening complete; residuals open* (STATUS §3, D2 row). `Connection` is live for PLAID only (`lib/plaid/exchangeToken.ts` writes it). `ProviderAccountIdentity` is backfilled + dual-written for PLAID; **WALLET is not** (`schema.prisma` §ProviderAccountIdentity comment; investigation `D2_STEP1C_C_WALLET_IDENTITY_COLLISION`). PLAID read cutover is complete with a **warn-and-fallback to the legacy `plaidAccountId` lookup**, whose removal is gated on a production observation window (D2 Step 7). The `plaidAdapter` (`lib/providers/plaid/adapter.ts`) is a "pure re-export, zero logic," **not referenced by any route**, and deliberately **not typed against a shared `ProviderAdapter` interface** — "out of scope until a second sync provider exists." The ProviderCatalog (`lib/providers/catalog.ts`, D6/D7) ships as a static registry with disabled placeholders for `coinbase-native`, `schwab-native`, `kraken-native`.

**What remaining provider work can complete before adding another provider?** Three self-contained items, none of which require a new provider to exist:

### Recommendation D2-a — WALLET provider-identity backfill + dual-write

- **Purpose:** Bring WALLET to identity-parity with PLAID: backfill `ProviderAccountIdentity` for existing wallet accounts and dual-write on wallet account creation, resolving the known identity-collision design (`D2_STEP1C_C`).
- **Why now:** WALLET is already an enabled catalog entry and an existing account type, but it's the one live provider without a canonical identity row — a latent inconsistency that grows with every wallet added. Closing it now means the identity layer is uniform *before* any native exchange/brokerage adapter builds on it.
- **Dependency graph:** Depends on the `ProviderAccountIdentity` model (exists) and the multi-account correction already in schema (D2 Step 1D, done). Blocks nothing; unblocks eventual legacy `walletAddress` column retirement.
- **Merge-conflict risk:** **Low** — confined to `lib/accounts/provider-identity.ts` and the wallet creation path; the collision investigation already exists as a spec. No overlap with FlowType.
- **Architectural payoff:** Uniform identity resolution across all live providers; removes a special-case; prerequisite for retiring ad hoc provider columns on `FinancialAccount`.
- **Estimated size:** **M** (backfill script + dual-write + tests; identity-collision handling is the real content).
- **Recommended order:** First D2 lane item.

### Recommendation D2-b — Formalize the `ProviderAdapter` interface around the existing Plaid seam

- **Purpose:** Give `plaidAdapter` a typed `ProviderAdapter` contract (`refreshItem`, `syncTransactions`, capability flags) and route one caller through it, so the seam is real rather than a re-export.
- **Why now:** The catalog already encodes `dispatch` routing and disabled native placeholders that *point at* this seam. Defining the interface now — with exactly one implementation — is the low-risk moment: no second provider yet means no behavioral divergence to reconcile, and it converts "add a provider = ad hoc functions" into "add a provider = implement an interface."
- **Dependency graph:** Depends on the existing `lib/plaid/*` functions (stable). Enables (does not block) the disabled catalog entries and any future sync provider.
- **Merge-conflict risk:** **Low-medium.** The interface file is new; wiring one route to it is a small, revertible change. Keep it additive (adapter beside direct calls) to stay behind the same revert boundary the current code already assumes.
- **Architectural payoff:** Turns the catalog's routing promise into an enforced contract; each future provider becomes confined-blast-radius work. Directly attacks Bottleneck #5.
- **Estimated size:** **M**.
- **Recommended order:** After D2-a; can run parallel to PE lane.

### Recommendation D2-c — Prepare (not execute) the PLAID legacy-fallback removal

- **Purpose:** Add the production observation instrumentation that Step 7 stabilization is *waiting on* — count how often the warn-and-fallback path fires — so the fallback's removal becomes a data-gated decision instead of an open-ended wait.
- **Why now:** The fallback is dead weight the moment the identity read is proven exhaustive; the only thing missing is the evidence to prove it. Instrumentation is additive and independent.
- **Dependency graph:** Depends on the existing identity read path. Gates the eventual `plaidAccountId`-lookup removal (a subtractive step — deferred per "keep changes additive before subtractive").
- **Merge-conflict risk:** **Very low** — logging/metric only.
- **Architectural payoff:** Converts a stalled subtractive cleanup into a measurable one; shrinks the dual-read surface over time.
- **Estimated size:** **S**.
- **Recommended order:** Any time; pairs naturally with D2-b.

## 3. Space infrastructure (backend, not UI)

**State (grounded).** Composition backend is real: `SpaceDashboardSection` (per-Space rows), `lib/space-presets.ts` (category → default sections, seeded on Space creation), and `lib/widget-registry.ts` (815 lines — "central source of truth for every section key," with data requirements, config schema, and status; the compositor looks entries up here, adding a section = one entry). Permissions exist as `SpaceMemberRole` (OWNER/ADMIN/MEMBER/VIEWER) + per-link `VisibilityLevel`, but role/lifecycle rules are **"enforced in the API layer, not the schema"** (`schema.prisma` §Space comment). `resolveSpaceContext` (`lib/space.ts`) is the membership gate.

**What infrastructure is still missing?**

- **Templates:** No `SpaceTemplate` model exists. D9 is **Parked** (STATUS §3). Presets are *hardcoded TypeScript*, not data. The D4 domain manifest already carries a `templateId` hook that is *accepted and ignored* (`lib/ai/domain-manifest.ts` §D9 hook) — the seam is drawn but empty.
- **Inheritance:** None. Categories map directly to preset arrays; there is no base→override chain, no template versioning, no user-defined templates.
- **Presets:** Exist but as code (`space-presets.ts`), so any change is a deploy, not a config edit.
- **Permissions:** No central policy/authorization module — role checks are re-implemented per route (see Bottleneck #6). This is real security-relevant debt.
- **Section registry / dashboard composition:** **Present and healthy** (`widget-registry.ts` + `SpaceDashboardSection`). This is the mature part.

### Recommendation SP-1 — SpaceTemplate foundation (D9), additive, data-behind-the-existing-seam

- **Purpose:** Introduce a `SpaceTemplate` model that backs the *already-declared* `templateId` manifest hook and the preset list — turning Space configuration from code into data, without changing any current behavior (existing Spaces keep resolving via category default).
- **Why now:** The consuming seam already exists and is documented in two places (`domain-manifest.ts`, `space-presets.ts` "optionally import from @prisma/client instead"). Building the model now, purely additive, means the AI context builder, presets, and any future template-driven Space all read one source instead of three. It is the backend prerequisite that must precede — and is independent of — any Space Dashboard UI work (which is excluded here).
- **Dependency graph:** Depends on nothing (additive table). *Enables* template inheritance, user-defined Spaces, the parked PublishedAccountView, and richer domain manifests. The manifest hook (`getDomainManifest(category, templateId)`) is the ready-made integration point.
- **Merge-conflict risk:** **Low.** New model + new module; the only touch to existing code is having `getDomainManifest` honor a non-null `templateId` (a branch the signature already reserves). No UI, no FlowType overlap. Note: D9 is currently *Parked*, so this needs an explicit un-park approval before schema work — but the investigation/checklist is lateral and can proceed now.
- **Architectural payoff:** Collapses three parallel "what belongs to this Space type" declarations (presets, manifest, widget defaults) toward one; unlocks inheritance and versioning; directly attacks Bottleneck #8 (config-as-code).
- **Estimated size:** **L** if implemented; **M** for the investigation + impact map + additive-schema checklist this audit recommends producing first.
- **Recommended order:** Start the investigation/checklist now (lateral, no code). Schema work gated on un-park approval.

### Recommendation SP-2 — Centralize Space authorization into a policy module

- **Purpose:** Extract the scattered API-layer role/lifecycle/visibility checks into one tested `canX(space, member, action)` policy module, called from routes rather than re-derived in each.
- **Why now:** Every new Space-scoped route today re-implements the OWNER/ADMIN/VIEWER + PERSONAL-Space-cannot-be-deleted + visibility rules by hand; the schema comment explicitly notes enforcement lives only in the API layer. This is a growing correctness-and-security surface (KD-1/KD-15/KD-19 were all "a read surface disagreeing with the canonical rule" — the same failure class, four times). A single policy home is the structural fix.
- **Dependency graph:** Depends on `resolveSpaceContext` (exists) and the visibility predicate (`lib/ai/visibility.ts`, exists). No dependents blocked; every future route benefits.
- **Merge-conflict risk:** **Medium** — refactor touches multiple route files, so sequence it away from concurrent route edits; the module itself is additive and can land before any callers migrate.
- **Architectural payoff:** Eliminates a recurring leak class; makes authorization auditable in one place; prerequisite for any shared-Space feature growth.
- **Estimated size:** **M** (module + incremental route migration).
- **Recommended order:** Module first (additive), migrate routes opportunistically. High leverage; can run parallel to everything.

## 4. AI architecture (ignoring model prompts)

**State (grounded).** The context stack is strong. `buildContext()` (`lib/ai/context-builder.ts`) does membership validation → AiAgent load → domain resolution (manifest ∩ agentScope ∩ scopeOverride) → parallel assembly with **per-domain error isolation** → signal detection → audit log. Registries mirror each other (`assembler-registry.ts`, `signals/registry.ts`, `context-priority/registry.ts`, PE `registry.ts`) — a consistent "pure function per key" spine. The **deterministic assessment engine** (AI-2, `lib/ai/intelligence/annotations.ts`) and **output validator** (AI-4, promoted from shadow to live `annotate` enforcement, KD-2 closed) are done. The **context-priority planner** (`lib/ai/context-priority/planner.ts`, D6.3D) is a *pure, shadow-mode* selection planner — it computes which sections *would* be trimmed under a token budget and why, but **does not yet mutate the prompt**.

- **Context builder:** Mature; the main gap is the empty D9 `templateId` seam (see SP-1).
- **Memory boundaries:** `AiAgent` has `agentScope` (a restriction layer) but **no persistent memory model**; `AiAdvice` is per-run history only, and per project rules `Conversation`/`Message` tables are forbidden. So there is *no conversation/memory substrate* — correctly deferred to AI-5 (v2.6a). Not a lateral target.
- **Deterministic preprocessing:** The planner is the asset. It is fully built and tested but **shadow-only**; promotion is AI-4's remaining item, gated to v2.6b.
- **Explainability:** Strong primitives exist — validator reconciliation, PE provenance, planner "why trimmed," attribution disclosure (KD-18). No unified explainability surface ties them together.
- **Orchestration:** The registries are the orchestration substrate; the missing piece is that the planner isn't wired to actually budget.

### Recommendation AI-a — Planner promotion *harness* (shadow→live scaffolding, default off)

- **Purpose:** Build the enforcement seam that lets `planContextSelection()` drive real serialization under a flag defaulting to shadow — the same shadow→live→default ladder AI-4's validator already used successfully (KD-2).
- **Why now:** The planner is *already pure and tested*; the only thing between it and value is a budgeting choke point. Building the harness (not flipping it) is lateral, mirrors a proven pattern, and de-risks the v2.6b promotion by letting it land as a flag flip rather than new architecture.
- **Dependency graph:** Depends on the planner (done) and the chat serialization path. Gated *for activation* to v2.6b, but the harness itself is independent.
- **Merge-conflict risk:** **Medium** — touches the chat serialization choke point, which is prompt-adjacent. Keep strictly to the section-selection boundary (no prompt-text edits) to avoid the FlowType/AI-5 prompt lanes.
- **Architectural payoff:** Turns a stranded shadow module into a switchable capability; establishes token-budget discipline as infrastructure rather than ad hoc truncation.
- **Estimated size:** **M**.
- **Recommended order:** After the D4 lane confirms no concurrent serialization edits; otherwise parallelizable. Lower priority than SP/PE/security items because activation is already milestone-gated.

*(Explainability unification and any memory substrate are deferred: the former has no clear consumer until a surface asks for it; the latter is AI-5/v2.6a and forbidden-table territory. Neither is a good lateral bet now.)*

## 5. Security — encryption architecture

**State (grounded).** D14 is **Complete** (STATUS §3): `lib/plaid/encryption.ts` implements AES-256-GCM with **HKDF-SHA-256 per-purpose subkey derivation** (Slices 1–4). A purpose registry isolates subkeys (`PLAID_ACCESS_TOKEN`, `TOTP_SECRET`, `DATE_OF_BIRTH`, `CONNECTION_CREDENTIAL`). Reads are dual-format: v2 (`v2:iv:tag:ct`, derived key) and legacy v1 (`iv:tag:ct`, root key). Password reset tokens are hashed at rest (D11, `lib/password-reset-token.ts`). Recovery codes are bcrypt-hashed.

**Remaining security debt that can complete independently:**

- **KD-6 — v1→v2 re-encryption (D14 Slice 5), Open, owned v2.5.** Legacy v1 root-key ciphertexts are *not* re-encrypted, so the deprecated `encrypt()`/`decrypt()` cannot be removed and the codebase carries two crypto paths indefinitely. This is the single cleanest lateral security task in the repo.
- **JWT session revocation.** `UserSession` is *informational only* — the schema comment states "JWT sessions cannot be invalidated mid-token-lifetime without a blocklist or short-lived tokens." Real revocation is unbuilt. Larger, auth-flow-adjacent; not as cleanly lateral as KD-6.

### Recommendation SEC-1 — KD-6 v1→v2 ciphertext re-encryption

- **Purpose:** Migrate all v1 root-key ciphertexts to v2 purpose-derived format (idempotent, dual-format-safe), then retire the deprecated `encrypt()`/`decrypt()` once no v1 rows remain.
- **Why now:** It is fully specified (Slice 5), self-contained, additive-then-subtractive within one subsystem, and every day it stays open the root key protects more fields than it should. Nothing else depends on it, and it depends on nothing in flight.
- **Dependency graph:** Depends only on the shipped HKDF machinery. The subtractive tail (removing the deprecated functions) is gated on a "zero v1 rows" verification — honoring "additive before subtractive."
- **Merge-conflict risk:** **Very low** — a migration script + call-site verification, no overlap with any active lane.
- **Architectural payoff:** Collapses two crypto formats to one; enables root-key rotation to become meaningful (today the root key still directly protects v1 data); removes a deprecated API.
- **Estimated size:** **M** (backfill script + verification + guarded removal of deprecated functions).
- **Recommended order:** **High-priority lateral item — start immediately.** It is the least entangled leverage in the whole audit.

## 6. Snapshot architecture

**State (grounded).** The only production writer is `regenerateSpaceSnapshot()` (`lib/snapshots/regenerate.ts`): it recomputes **today's** `SpaceSnapshot` row from *current* balances (`getAccounts` + `classifyAccounts`) and upserts on `[spaceId, date]`. It is wired into `lib/plaid/refresh.ts` (manual Refresh) and fans out to every Space an account is shared into. Charts read **exclusively** from `SpaceSnapshot` (`lib/data/snapshots.ts`). Before `regenerate.ts` existed, only `seed.ts` ever wrote snapshots, so real Spaces rendered blank charts.

**The gaps are structural:**

- **Generation coverage:** Snapshots are written *on balance-change events (refresh)* only. The daily cron that should guarantee a row per Space per day is **not running** — `jobs/take-snapshot.ts` is an empty stub (`export {}`), and `startScheduler()` is never invoked (Area 8). So a Space that isn't refreshed gets *gaps* in its time series.
- **Historical reconstruction:** **None.** There is no backfill from transaction history; a newly connected account starts its net-worth chart at day zero. This is exactly what `INITIAL_DAILYSNAPSHOT_BACKFILL_INVESTIGATION.md` and the D2.x charter Phase 4 ("30-day `SpaceSnapshot` backfill after history completes") anticipate.
- **Caching:** The pre-computed row *is* the cache (derived fields `netWorth`/`totalAssets`/`netLiquid`/`cashOnHand` are materialized for fast chart reads) — this part is sound.
- **Incremental snapshots:** None — every write is a full recompute-and-upsert. Fine at current scale; a future concern, not a current one.

### Recommendation SN-1 — Historical snapshot backfill (deterministic reconstruction)

- **Purpose:** Reconstruct missing `SpaceSnapshot` rows for a date range from available history (transaction-derived balances where possible; carry-forward where not), producing a continuous series instead of one that starts at connection time.
- **Why now:** It is the acknowledged prerequisite for the D2.x flagship *and* for the Net Worth perspective (PE-2) and any trend-based AI answer. The investigation already exists. It is pure backend math over existing rows — no UI, no FlowType semantics (it uses balances/classification, not flow classification).
- **Dependency graph:** Depends on `regenerateSpaceSnapshot` (exists) and transaction history. *Enables* PE-2, D2.x Phase 4, and richer snapshot-history AI context. Sequenced by STATUS after D2.x — but the reconstruction engine is buildable and testable independently of the sync-UX orchestration that consumes it.
- **Merge-conflict risk:** **Low** — a new module beside `regenerate.ts`; the idempotent upsert contract already exists.
- **Architectural payoff:** Turns the snapshot series from best-effort into complete; every time-series consumer (charts, PE, AI trends) inherits the fix once.
- **Estimated size:** **M–L** (reconstruction math + carry-forward rules + idempotency/backfill tests).
- **Recommended order:** After SN-2 wiring (so backfilled rows aren't immediately re-gapped), or jointly. High leverage.

### Recommendation SN-2 — Daily snapshot generation (see JOB-1 — same root cause)

The daily-coverage gap is not really a snapshot problem; it is the **job-execution** problem in Area 8. `take-snapshot` is a stub and the scheduler never starts. Fixing it lives in JOB-1 below; it is called out here because Snapshot integrity *depends* on it.

## 7. Event architecture

**State (grounded).** There is **no event model of any kind.** A repository-wide search for `emit(`, `EventEmitter`, `dispatchEvent`, `eventBus`, `publish(` returns nothing in `lib/`, `app/`, or `jobs/`. Cross-subsystem coordination is hardcoded direct function calls: `lib/plaid/refresh.ts` calls `regenerateSnapshotsForAccounts()` directly; reconcile, sync, and snapshot regen invoke each other inline. `AuditLog` is the closest thing to an event stream, but it is a write-only side-effect log, not a subscribable bus.

**Should Fourth Meridian begin introducing an event model? — Justified, but as a small seam, not a framework.** The examples in the prompt (`AccountConnected`, `TransactionImported`, `SnapshotCreated`, `GoalCompleted`) are *exactly* the moments where today a caller must know every downstream reaction. The pressure is real and growing: every new reaction to "an account synced" means editing `refresh.ts` again (see Bottleneck #3). But a full event-sourcing / outbox framework is over-scoped for the current size, and the project rules forbid speculative table growth.

### Recommendation EV-1 — Investigation + minimal typed domain-event seam (investigation first)

- **Purpose:** Produce a decision record on whether to introduce a lightweight, *in-process, synchronous* typed domain-event emitter (a registry of `(eventName) → handlers`, mirroring the existing assembler/signal/lens registry pattern the codebase already trusts) — starting with `SnapshotCreated`/`AccountConnected` as the first two events, replacing the hardcoded `refresh → regenerate` call.
- **Why now:** The codebase already has five "register a pure function per key" registries; a synchronous event registry is the *same* proven shape, not a new mental model. Doing it while there are only ~3 coordination edges keeps the migration trivial; every quarter of delay adds more hardcoded fan-out to untangle. Starting with an *investigation* (no code) is the correct lateral first step and respects "only if justified."
- **Dependency graph:** No dependencies. *Enables* decoupled reactions for Snapshot (Area 6), background jobs (Area 8), and future notifications (deferred). Deliberately **not** persisted (no outbox/table) in v1 — synchronous, in-process, no new schema — to stay inside the additive/no-speculative-tables rules.
- **Merge-conflict risk:** **Low** — a new registry module; the first migration edits one call site (`refresh.ts`).
- **Architectural payoff:** Converts N-caller-knows-all-reactions into publish/subscribe; the single highest-leverage decoupling available before the subsystem count grows further (Bottleneck #3).
- **Estimated size:** **S** for the investigation/decision record (recommended now); **M** for the seam + first-two-events migration (after approval).
- **Recommended order:** Investigation now; implementation gated on the decision record and explicit approval (it's greenfield architecture, so it deserves a freeze-doc-style sign-off).

## 8. Background job architecture

**State (grounded).** This is the **weakest substrate in the system.** `jobs/scheduler.ts` uses `setInterval`, and its own header states `startScheduler()` **is never invoked** (no `instrumentation.ts` hook). `jobs/take-snapshot.ts` is an empty stub. `run-ai-advice.ts` and `sync-crypto.ts` are stubs (STATUS D5). `AiAdvice` has **never had a write path** (KD-14). Only `sync-banks` has a real trigger — a Vercel cron hitting `app/api/jobs/sync-banks/route.ts` (`vercel.json`). There is **no queue** — no BullMQ/Redis/ioredis anywhere; the header itself says "Replace with node-cron or BullMQ for production resilience." `purgeTrash` is registered but never started. Long-running imports (D2.x) are planned to use `after()`/`waitUntil` with "no queue/`SyncJob` yet."

This is tracked as **D5 (Active, reopened)** and **KD-14 (High, blocks v2.6b)**.

### Recommendation JOB-1 — Cron-triggered job substrate (align all jobs to the working `sync-banks` pattern)

- **Purpose:** Give every scheduled job the *one* execution model that already works in production — a Vercel-cron-hit API route — instead of the dead `setInterval` scheduler. Concretely: implement `take-snapshot` and wire it (+ `purge-trash`) to cron routes mirroring `app/api/jobs/sync-banks/route.ts`; leave `run-ai-advice`/`AiAdvice` write path explicitly to v2.6b.
- **Why now:** Daily snapshot coverage (Area 6), trash purging (D8), and eventually AI advice all silently do nothing today. The working pattern already exists and is proven; this is *alignment*, not invention. It is the substrate that Areas 6 and (partly) 4 depend on.
- **Dependency graph:** `take-snapshot` depends on `regenerateSpaceSnapshot` (exists). Snapshot completeness (SN-1/SN-2) depends on this. `AiAdvice` write path is intentionally out (v2.6b, and touches AI generation — adjacent to excluded lanes).
- **Merge-conflict risk:** **Low** — new API routes + stub implementations + `vercel.json` cron entries; no overlap with FlowType or UI.
- **Architectural payoff:** Converts "registered but never runs" into a real, observable execution tier; removes the dead `setInterval` scheduler as the aspirational-but-inert path; unblocks Snapshot integrity.
- **Estimated size:** **M** (implement `take-snapshot`, 2–3 cron routes, cron config, idempotency guards).
- **Recommended order:** **High priority** — it is the dependency under Area 6 and a standing correctness gap. Do JOB-1 before SN-1/SN-2 rely on it.

### Recommendation JOB-2 — Decision record on a durable queue / `SyncJob` model

- **Purpose:** Decide *whether/when* a real queue (`SyncJob` model, or BullMQ/Redis) is warranted, versus continuing with `after()`/`waitUntil` + cron fallback that D2.x assumes.
- **Why now:** D2.x explicitly defers `SyncJob`; that deferral should be a recorded decision with trigger conditions (e.g. "when import volume or retry needs exceed cron/`after()`"), not an implicit gap. Cheap to write now, prevents a premature build.
- **Dependency graph:** Informs D2.x Phase design and JOB-1's ceiling. No code.
- **Merge-conflict risk:** **None** (investigation).
- **Architectural payoff:** Prevents both premature queue-building and surprise scaling walls; names the trigger.
- **Estimated size:** **S** (decision record).
- **Recommended order:** Any time; low urgency but high clarity value.

## 9. Search architecture

**State (grounded).** There is **no unified search/index layer.** No full-text/`tsvector`/GIN/Elastic/Meili anywhere in `lib`, `app`, `prisma`; the "search" hits are admin-table UI filters (`app/admin/*`) only. Today's entity counts are small and lookups are direct indexed Prisma queries (`plaidTransactionId @unique`, `plaidAccountId @unique`, `SpaceAccountLink` composite indexes, etc.), which are correct and sufficient at current scale.

**Should a unified search/index exist before documents, merchants, providers, frameworks, and spaces grow further? — Not yet; but reserve the seam.** Building a cross-entity index now would be speculative infrastructure against entities (documents, frameworks) that **don't exist as models yet** — a violation of the additive/no-speculative-build discipline. The honest finding: search is a *future* bottleneck (ranked below), not a *current* lateral opportunity. The one useful lateral act is a short decision record fixing the trigger condition and the likely shape (Postgres FTS first, given the existing single-Postgres footprint — no new infra), so that when merchants/documents land, the team reaches for one index rather than N ad hoc `contains` filters.

### Recommendation SR-1 — Search decision record only (no build)

- **Purpose:** Record when a unified index becomes warranted (first cross-entity user-facing search, or first free-text entity like documents), and pre-commit to Postgres-native FTS to avoid new infrastructure.
- **Why now:** Cheap; prevents both premature building and a future scramble; the "before entities grow further" framing in the prompt is best answered by a written trigger, not code.
- **Merge-conflict risk / size:** **None / S** (investigation).
- **Recommended order:** Low priority; write alongside the Event and Queue decision records as a batch of "reserve-the-seam" docs.

# Part B — The 10 Largest Architectural Bottlenecks (ranked by urgency)

Ranking weighs: how much *current* code pays the tax, how fast the tax compounds, and how entangled removal becomes if deferred. FlowType/Merchant/Atlas/Brief/Dashboard-UI concerns are excluded by mandate.

**1. Dual-anchor account model (legacy `Account` ⟷ `FinancialAccount`; dual FK on `Transaction`/`Holding`).** *Most pervasive.* `Transaction.accountId`/`financialAccountId` and `Holding.accountId`/`financialAccountId` are both optional with "exactly one set"; every read normalizes both (`lib/data/accounts.ts`, D11 comments). Legacy `Account` is deliberately retained (D3 closure note; retirement "gated on prod Phase 0 counts + explicit approval"). Every new reader must know the dual shape. Urgency high because *every* data-layer change pays it — but removal is subtractive and correctly gated, so the lateral move is **instrumentation toward retirement (D2-c-style counts), not removal.**

**2. Background-job execution substrate (D5/KD-14).** Nothing scheduled actually runs: `startScheduler()` never invoked, `take-snapshot` a stub, `AiAdvice` never written. Directly starves Snapshot integrity and ambient intelligence. Urgent because it's not debt-you'll-pay-later — it's capability that silently doesn't exist today. **Addressed by JOB-1.**

**3. No event/decoupling layer.** Cross-subsystem reactions are hardcoded caller-knows-all-downstream (`refresh → regenerateSnapshotsForAccounts`). Compounds with every subsystem added; cheapest to fix while edges are few. **Addressed by EV-1.**

**4. Snapshot generation coverage + no historical reconstruction.** Time series has gaps (event-only writes) and starts at connection time (no backfill). Every trend/chart/net-worth answer inherits the gap. **Addressed by SN-1 + JOB-1.**

**5. Provider adapter interface not formalized.** Everything is Plaid-shaped; `plaidAdapter` is a re-export with no contract; native placeholders are disabled. A second sync provider currently means ad hoc functions, not an interface implementation. **Addressed by D2-b.**

**6. Authorization scattered in the API layer.** Role/lifecycle/visibility rules re-implemented per route (schema comment; four KD leaks — KD-1/15/19 + class). No central policy engine. Security-relevant and recurring. **Addressed by SP-2.**

**7. Money represented as `Float`, USD-assumed (MC1).** Every balance/amount/snapshot field is `Float` with `currency` defaulting to `"USD"`; aggregation chokepoints (`sumBalances`/`classifyAccounts`) assume one currency. MC1 is *Approved, foundational, planned* — explicitly "future provider integrations depend on it and should build on it rather than precede it." Float-for-money is a correctness bottleneck independent of currency. High long-term urgency; sequenced after D2.x + snapshot backfill, so **not a v2.5 lateral start** — but its *provenance-stamping Phase 0* (additive, behavior-neutral, "cannot be reconstructed later") is the one piece that genuinely shouldn't wait.

**8. Space configuration as code, not data (D9 parked).** Presets, domain manifest, and widget defaults are three parallel hardcoded declarations of "what belongs to this Space type"; no `SpaceTemplate`, no inheritance, no versioning. Blocks user-defined Spaces and template evolution. **Addressed by SP-1.**

**9. Encryption format duality + no meaningful key rotation (KD-6).** Two crypto paths (v1 root-key, v2 derived) coexist because Slice 5 re-encryption is unbuilt; the deprecated API can't be removed and root-key rotation can't fully protect v1 data. Cleanly self-contained. **Addressed by SEC-1.**

**10. No search/index layer as entities multiply.** Currently fine (small, indexed); becomes a bottleneck the moment free-text entities (documents, frameworks) or cross-entity search arrive. A *future* pressure point — reserve the seam, don't build. **Addressed by SR-1 (decision record).**

*Just-outside-the-ten, noted:* JWT session non-revocation (`UserSession` informational only) — real security debt, but auth-flow-entangled rather than cleanly lateral; audit-log write amplification (KD-12, Low); the legacy `plaidAccountId` fallback read (shrinking, gated).

# Part C — Recommended parallel execution order

These lanes are mutually low-conflict and independent of FlowType. Ordered by leverage-per-risk:

1. **SEC-1 (KD-6 re-encryption)** — start immediately; least entangled, self-contained security leverage.
2. **JOB-1 (cron job substrate)** — high priority; unblocks Snapshot integrity and removes a standing "nothing runs" gap.
3. **SP-2 (central authorization module)** — additive module now, migrate routes opportunistically; kills a recurring leak class.
4. **PE-2 (Net Worth lens)** then **PE-3 (holdings extraction + Investments lens)** — cheapest AI-adjacent leverage; exercises snapshot + holdings reads.
5. **D2-a (WALLET identity)** → **D2-b (adapter interface)** → **D2-c (fallback instrumentation)** — one contained provider lane.
6. **SN-1 (historical backfill)** — after JOB-1 so backfilled rows aren't re-gapped.
7. **Investigations/decision records (no code, batchable now):** EV-1 (event seam), SP-1 (SpaceTemplate/D9 un-park), JOB-2 (queue trigger), SR-1 (search trigger), plus MC1 Phase 0 provenance checklist (the one MC1 piece that shouldn't wait).

**Sequencing note per project rules:** every item above is additive-before-subtractive and warrants its own impact map, rollback plan, and validation checklist (`prisma generate` / `migrate dev` if schema changes / `tsc --noEmit` / `lint` / targeted tests) before any code — matching the established Phase 2 working style. Greenfield items (EV-1, SP-1/D9) additionally need explicit un-park/architecture approval, since D9 and D12 are currently Parked and an event model is new surface.
