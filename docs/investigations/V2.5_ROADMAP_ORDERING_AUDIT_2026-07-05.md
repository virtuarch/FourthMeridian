# v2.5 Roadmap Ordering Audit — 2026-07-05

**Status:** Investigation only. No implementation. Audited against the working tree on `feature/v2.5-spaces-completion` at `fcadc2a` (D-TEST), which is 17 commits past the `2db97ef` checkpoint STATUS.md §3 was last verified against.

---

## 1. Does the proposed ordering (Merchant → FlowType → AiAdvice → Catalogs → SP-1) still make sense?

**Directionally yes; factually no — because two of the three "future" items are already mid-flight, and one premise is inverted relative to the architecture as built.**

### 1.1 FlowType is not a candidate; it is a half-finished cutover

Per `docs/initiatives/flowtype/P5_END_TO_END_CUTOVER_STATE_INVESTIGATION.md` (2026-07-04), verified against source today:

- **Write side: complete.** All three production writers (Plaid sync, import route, seed) populate `flowType` via `classifyFlow`/`buildFlowWriteFields`. Schema (enum + nullable columns + rollup indexes, incl. `counterpartyAccountId`) applied in `20260703130000_v255_flowtype_schema`. Backfill script exists (`scripts/backfill-flowtype.ts`).
- **Read side: ~40%.** Banking + Space panel chips read `flowType` (Slices 0–2 done). The AI assembler, annotations gate, chat serializer, and `DebtClient` still run on category + amount sign (Slices 3–7 not started).

This means the system currently has **two live semantic authorities**: the dashboard computes Spend/In from `flowType` while every AI number and the Daily Brief compute from category+sign. They disagree by construction (the Slice-2 income-inflation fix already changed dashboard numbers; AI numbers haven't followed). Leaving this seam open while building Merchant Intelligence's persisted tier would be the exact "new code built on two sources of truth" failure the 2026-07-02 roadmap revision warned about.

### 1.2 Merchant Intelligence Slice 1 already shipped

Commits `c8601b9` / `275a9c8` landed the curated global rule catalog (`lib/transactions/merchant-rules.ts`) wired **above PFC** inside `mapPlaidCategory` (`lib/transactions/plaid-category.ts:96,116`), plus `scripts/backfill-merchant-categories.ts`. The pure normalizer (`lib/transactions/merchant.ts`) and the dedup normalizer (`fingerprint.ts`) predate it. What does **not** exist: a Merchant table/graph, `merchantId` FK (only the `merchantEntityId` forward-seed, written and never read), user/Space overrides, category provenance (`categorySource`), and cadence/recurring detection. So "Merchant Intelligence" as a v2.5 item is really the **persisted tier** — and it is gated on a decision, not code: the `TransactionCategory` enum gap (no Medical/Education/Auto/Transport/Entertainment — investigation §4.4 calls it "a binding constraint on any merchant layer").

### 1.3 The dependency premise is inverted

The roadmap reasoning says "FlowType should consume normalized merchants rather than raw descriptions." The architecture as built — deliberately — says the opposite chain: **merchant rules → category → flow-classifier**. `flow-classifier.ts` is the single flow authority; merchant text is an available input it intentionally does not use, and `MERCHANT_INTELLIGENCE_LAYER_INVESTIGATION.md` §5 codifies "do not let a merchant rule write flowType directly." FlowType's remaining work (read cutover) needs **nothing** from the merchant persisted tier. Durable merchant identity is not a prerequisite for durable flow semantics; the `merchantEntityId` seed allows the Merchant table to backfill later without a Plaid re-fetch.

### 1.4 Recommended v2.5 order

1. **FlowType P5 Slices 3–7** (finish the read cutover). Code-only, each slice independently revertible, already planned and investigated. Slice 3 delivers the KD-18 per-liability capability; Slice 4 is the one behavior slice and needs a numbers sign-off, nothing else.
2. **v2.5 seam/debt closure riding alongside:** legacy `Account` out of the 3 remaining read sites (`lib/imports/authorize.ts`, `app/api/admin/overview/route.ts`, `app/api/accounts/[id]/transactions/route.ts` — an unmet v2.5 exit criterion), `.env.example` flags, observability counters, KD-13 dirs.
3. **Merchant Intelligence persisted tier**, opened by the Slice 1a **category-enum decision**, then user overrides (first schema: `MerchantRule` + `categorySource` provenance), then cadence detection as its own slice.
4. **AiAdvice minimal writer + surfacing** (see §5 — it does not need the scheduler).
5. **Provider Catalogs: agree, defer.** D6's own resume trigger ("a second provider or launch UI needs it") hasn't fired, and MC1's charter explicitly sequences currency architecture **before** provider expansion.
6. **SP-1 SpaceTemplate: keep parked.** No `SpaceTemplate` model exists (only a forward-looking comment at `schema.prisma:476`); D9 is parked in STATUS §8 with an explicit unpark condition (real users requesting templates), and the 2026-07 refresh audit independently down-ranked it as "zero user value unlocking a deferred marketplace."

## 3. Missing from v2.5 (architectural necessities, before Ambient Intelligence)

1. **MC1 Phase 0 — currency provenance stamping.** Approved, additive, behavior-neutral, and the charter's own words: "provenance cannot be reconstructed later." Every merchant/flow/advice row written between now and Phase 0 is another row that can't be stamped retroactively. This belongs in the v2.5 window even though full MC1 doesn't.
2. **The money-type decision (Float → Decimal/integer-cents).** Zero `Decimal` in the schema today. The 07-refresh audit ranked it the #1 latent defect; §8 parks the migration as "plan during v2.5; execute post-v2.6b." AiAdvice will *persist* derived figures — persisting advice computed from drifting floats before the model is fixed bakes the imprecision into user-visible history. At minimum the plan must exist before the writer ships.
3. **Job substrate (D5/JOB-1).** Ambient Intelligence's entry criteria literally cannot be met without it: `startScheduler()` has no invocation point, two of four jobs are stubs. The D2.x-deferred SyncJob/webhook/retry work should fold into the same substrate rather than be built twice.
4. **Category provenance + enum expansion decision (Merchant Slice 1a).** Without `categorySource` and the enum gap resolved, user overrides have no home and pattern rules (pharmacy/clinic/transit) have nowhere accurate to map. This is the true entry gate to the merchant persisted tier.
5. **Verification-debt closure** (§2 list). The approved ladder makes v2.4.5's exit the entry criteria for everything after; it is still open.
6. **Data-quality signal consumption.** `SyncIssue` (M1/M2) exists flag-only. Before advice generation, something must decide "is this Space's data healthy enough to speak from" — otherwise Ambient fires on incomplete syncs. The snapshot `quality` column and AI sync-health consumption are deferred to v2.6b; the *contract* (advice writer checks SyncIssue) should be designed with the writer.

## 4. Should Merchant Intelligence be a standalone subsystem?

**Yes — and it already is one, embryonically** (`lib/transactions/merchant.ts`, `merchant-rules.ts`, pure, dependency-free, tested). Preserve those boundaries as it grows:

**Inside Merchant Intelligence:** merchant identity (normalization, canonical key/name, future Merchant table + `merchantId` backfill via `merchantEntityId`), alias/rule catalog (global, locale-tagged regional, pattern families), the override precedence chain (user override → Space override → personal rule → global rule → PFC fallback → sign default), category *suggestion* output, and — as a sibling track, never folded in — cadence/recurring detection (time-series over the ledger; the PlayStation/Prime-Video case proves merchant name alone can't distinguish subscription from one-off).

**Not inside it:** flow semantics. Merchant Intelligence outputs **category and identity only**. `flow-classifier.ts` stays the single flow authority, consuming category + PFC + sign + account context. Even flow-shaped merchants (card payment, annual fee) route through category, per the investigation. Also not inside it: visibility (SAL) — a Space override is a categorization concept needing its own store, not a `SpaceAccountLink` extension.

**FlowType owns:** economic meaning per row — `flowType`, `flowDirection`, counterparty attribution (`counterpartyAccountId`, the KD-18 seam), confidence, versioned reclassification.

**AiAdvice owns:** persistence and surfacing of conclusions computed from facts the other two layers settled. `computeAssessment` is already pure and deterministic; the writer must serialize it, never re-derive transaction semantics. If the advice layer ever contains a "is this a transfer?" heuristic, a boundary has failed.

## 5. Hidden dependencies likely to change ordering

- **FlowType Slice 4 is gated on a human decision** (behavior sign-off: dividends→income, `Fee` reachable, refund netting), not engineering. Schedule the sign-off, or Slices 5–7 stall behind it.
- **KD-18's real capability arrives with FlowType Slice 3, and Slice 6 relaxes the guardrail.** Any AiAdvice work that mentions per-liability payments before Slice 3/6 will reproduce the fabricated-attribution defect the guardrail exists to prevent. FlowType-before-AiAdvice is therefore a hard edge, as suspected.
- **The AiAdvice writer does not depend on the scheduler.** `INVESTIGATION_AIADVICE_WRITER_SURFACING_LOOP.md` shapes it as a trigger-agnostic `writeAdviceForSpace()` — it can ship in v2.5 behind an on-demand/request trigger, with D5 supplying cadence later. This decouples "AiAdvice third" from the v2.6b scheduler work. Note the approved ladder's constraint: unprompted *delivery* still gates on v2.6a (AI-5); a persisted row surfaced when the user opens the Brief does not violate it.
- **Your candidate list omits AI-5 (v2.6a) entirely.** The approved roadmap puts Advisor Intelligence *between* financial-intelligence work and Ambient; Ambient's entry criteria reference it. Any v2.5 ordering that treats Ambient as "next after AiAdvice" skips an approved gate.
- **Merchant persisted tier depends on the enum decision (Slice 1a) and on `categorySource` provenance**; overrides also interact with backfills (a re-run must not clobber a user's correction — the `classifierVersion` pattern is the model).
- **MC1 precedes provider expansion by charter** — another reason Provider Catalogs stay parked.
- **AiAdvice model gaps surface at writer time:** append-only with no dedupe key, no `supersededAt`, no freshness column, and a regex-parsed markdown contract in `AdviceBanner`. The writer slice must add a dedupe/freshness mechanism or the Brief will spam identical advice rows.

## 6. Three highest-leverage investments (from the repository, not the roadmap)

1. **Finish the FlowType read cutover (P5 Slices 3–7).** The repo today contains two disagreeing semantic engines: dashboard chips on `flowType`, all 9.5k lines of AI + the Daily Brief on category+sign. The write side, backfill, indexes, tests, and slice plan already exist — this is the cheapest remaining distance between "platform" and "financial intelligence," and every other v2.5 item (merchant tier, advice writer, KD-18 capability) reads through it. Evidence: `P5_END_TO_END_CUTOVER_STATE_INVESTIGATION.md` §2.2 (0 `flowType` refs in the assembler) vs `BankingClient.tsx:170-177`.
2. **Money correctness: MC1 Phase 0 provenance + the Float→Decimal/cents plan.** Every monetary column is `Float`; provenance is unstampable retroactively; the project is pre-production, which is the last cheap moment. A financial-intelligence layer built on drifting substrate is polish on imprecise numbers — and AiAdvice persistence makes the imprecision permanent and user-visible. Evidence: zero `Decimal` in `schema.prisma`; MC1 charter's own sequencing doctrine.
3. **Operational substrate + one promoted insight: D5 scheduler entrypoint + the AiAdvice write path (KD-14).** The largest built asset in the repo — the deterministic assessment engine — computes on every chat and persists nothing; `run-ai-advice.ts` is `export {}`; the UI advertises runs that never happen. The platform→intelligence transition is, concretely, this one loop: assessment → `AiAdvice` row → Brief. The minimal writer needs no scheduler; the scheduler entrypoint then upgrades it from on-demand to ambient and simultaneously satisfies v2.6b's entry criteria. Evidence: `INVESTIGATION_AIADVICE_WRITER_SURFACING_LOOP.md` §2; `jobs/scheduler.ts` header.

**Summary recommendation:** v2.5 = FlowType completion + seam/debt closure + MC1 Phase 0 + merchant enum/override foundations + a scheduler-free AiAdvice writer. Provider Catalogs wait for their trigger; SP-1 stays parked; Ambient stays behind AI-5 and the job substrate, exactly as the approved ladder already says.
