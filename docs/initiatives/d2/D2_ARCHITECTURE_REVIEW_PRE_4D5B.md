# D2 Architecture Review — Before 4D-5b

Review only. No code, schema, migration, or roadmap-doc changes made. Grounded against the current state of `lib/imports/csv.ts`, `lib/imports/excel.ts`, `lib/transactions/fingerprint.ts`, `prisma/schema.prisma` (`Connection`, `ImportBatch`, `Transaction`, `FinancialAccount`), `docs/architecture/PHASE_2_DECISION_MATRIX.md`, `docs/initiatives/d2/D2_ROADMAP.md`, and the two prior planning docs this review builds on rather than repeats: `D2_IMPORT_MAPPING_LAYER_ROADMAP_PLACEMENT.md` ("Part B") and `D2_STEP4D_SEQUENCING_REVISION_PROPOSAL.md`.

Where Part B and the sequencing proposal already answered a question well, this review affirms and cites rather than re-deriving from scratch. Where reading the actual code surfaced something neither prior doc caught, that's called out explicitly — that's most of the value of this pass.

## Verdict, up front

The roadmap is sound and shouldn't change again: 4D-5b before 4D-4 QuickBooks, for the reasons the sequencing proposal already gave. The team's discipline about not persisting, not building UI, and not widening `HEADER_ALIASES` ahead of a concrete need is real and is the right instinct — this is not an over-engineered pipeline. But there is one assumption already on the table — the "header signature hash" as the lookup key for a saved mapping profile — that is the wrong primitive and will quietly break the first time a bank adds one column to its export. That should be corrected in 4D-5b's checklist before `ImportMappingProfile`'s schema is written, not discovered after. There's also a real, small case of duplicated validation logic in `csv.ts` today, and a real gap in what `ImportBatch` records about its own column resolution — both worth folding into 4D-5b's scope since 4D-5b is about to touch this exact code a third way.

## 1. Should the next step be 4D-5b instead of QuickBooks?

Yes. The sequencing proposal's case is correct and doesn't need re-litigating: QuickBooks's real differentiator is the match/classify stage (a durable `TxnID`, update-on-match semantics), not column shape — so mapping and QuickBooks are parallel concerns, not a dependency chain, and ordering should be decided on simplicity grounds. Mapping first is simpler because QuickBooks's own export-shape heterogeneity (Desktop vs. Online vs. report type) means it would otherwise need several hardcoded alias lists with no guarantee of completeness, exactly the problem the mapping layer exists to solve once.

One addition: don't treat 4D-5b as "implement what Part B already specified." Part B's persistence design (§3/§4 of that doc) was written before any of this code existed. Section 5 below explains why its specific matching mechanism needs to change before it becomes a schema.

## 2. Is anything architecturally missing before persisting mapping profiles?

Yes, one concrete gap: nothing today records which column resolution actually produced a given `ImportBatch`. `detectColumns()`, and now `applyExplicitMapping()`, both compute a `CsvColumnMap` and immediately discard it after `normalizeRow()`/`normalizeExcelRow()` consume it — `ImportBatch` has no field holding it. That's a minor gap today (there's exactly one possible source: auto-detect, or explicit-mapping-which-is-never-saved-anywhere-else). It becomes a real gap once 4D-5b adds a saved profile as a third source and 4D-5c adds fuzzy-suggestion as a fourth: at that point, "why did row 47 get parsed this way" has no answer anywhere in the data model, not even by looking at the `ImportBatch` row. Recommend an additive `ImportBatch.resolvedColumnMapping: Json?`, written once at batch-creation time regardless of which resolver produced it. This also sidesteps a provenance problem 4D-5b's own design would otherwise create: if `ImportBatch.mappingProfileId` is the only record of "what mapping was used" and a profile is later edited, every historical batch's effective mapping silently changes retroactively from a reporting standpoint, even though the actual `Transaction` rows were created under the old mapping. Snapshotting the resolved map at creation time avoids that.

Beyond that one gap, no — Part B's other groundwork (Space-scoping, additive-only schema, `detectColumns()` as the unconditional fast path) is the right shape and doesn't need revisiting.

## 3. Are we accidentally solving problems twice anywhere?

One real instance, small but worth fixing in 4D-5b: `detectColumns()` and `applyExplicitMapping()` independently implement the identical three-rule validation (date required; merchant-or-description required; amount-or-debit/credit required) with two separate sets of hand-written error strings. They operate on the same input shape (`string[]` headers resolving into the same `CsvColumnMap` accumulator) — this isn't a case like CSV-vs-Excel row normalization, where the duplication is justified by a genuine difference in underlying data (typed Excel cells vs. raw strings, documented explicitly in both files' headers). Here, the two functions differ only in *how* a field gets resolved (alias lookup vs. caller-supplied value), not in what "valid" means once resolved — that's exactly the kind of logic a shared `validateResolvedColumns(resolved): { error: string } | null` should own once. It's low-urgency today (six duplicated lines), but 4D-5b is about to add a third producer of `CsvColumnMap` (profile-driven resolution) that would otherwise hand-duplicate the same three checks a third time — worth folding the extraction into 4D-5b's own diff rather than letting it become a fourth copy later.

Everywhere else, duplication that looks superficially similar is actually justified and shouldn't be merged: CSV's `normalizeRow()` and Excel's `normalizeExcelRow()` share the same field-level logic but operate on genuinely different cell representations (a Date object or serial number vs. a date string; a typed number vs. a currency-formatted string) — collapsing them would mean threading typed-cell-awareness through `csv.ts`, which is exactly the "forced shared interface" the module's own header comment already declined to build. The account-level fingerprint in `reconcile.ts` and the transaction-level fingerprint in `fingerprint.ts` were already evaluated for unification during 4C and correctly left separate — they key on disjoint field sets (account identity vs. date+amount+merchant) and solve different problems that happen to share the word "fingerprint."

## 4. Is there another abstraction that should exist before `ImportMappingProfile` becomes a Prisma model?

Yes, two, both small. First, the validation extraction from §3. Second — and more important — the column-resolution call site itself is duplicated at the *call* level, not just within `csv.ts`: `route.ts`'s CSV branch and `excel.ts`'s `parseExcelFile()` each independently write the same `explicitMapping ? applyExplicitMapping(...) : detectColumns(...)` ternary. 4D-5a already had to touch both call sites once to add the second branch; 4D-5b adding a third source (saved profile lookup) and 4D-5c adding a fourth (fuzzy suggestion) means editing both call sites again, twice more, unless resolution is centralized first. Recommend a single `resolveColumns(headers, { explicitMapping?, savedProfiles? })` entry point in `csv.ts` that both call sites delegate to — Part B's own §8 already proposed this shape ("a new resolution function... that `detectColumns()` becomes the first branch of"); it just hasn't been built yet, and 4D-5a shipped without it because a two-way ternary didn't yet justify the extraction. A three-or-four-way one does.

Neither of these is schema work — both belong in `lib/imports/csv.ts` and should land as part of 4D-5b's implementation, before or alongside the `ImportMappingProfile` table, not as separate slices.

## 5. How would you fingerprint an import format?

Not the way Part B's §4 proposes, and this is the most consequential finding in this review. Part B recommends keying a saved profile by "the file's normalized column headers, sorted and hashed" — i.e., a hash of the *entire* header set. That's the wrong primitive: it's an equality check on something that's expected to drift (the full set of columns a bank chooses to export) when what actually matters is something much narrower (whether the columns a *mapping cares about* are still present).

The better fingerprint is a subset/dependency check, not a full-set hash: does this file's header list contain every header the profile's own mapping references? That's not a new algorithm to design — it's exactly what `applyExplicitMapping(headers, profile.mapping)` already computes. A saved profile, in this framing, is just a remembered `mapping` argument; "does this profile still apply to this file" and "apply this profile to this file" collapse into the same function call, with no separate signature-comparison step to keep in sync with it. This also means `ImportMappingProfile`'s schema doesn't need to invent a `headerSignature` column's hashing scheme at all going in — it needs a `mapping: Json` column shaped exactly like `applyExplicitMapping()`'s second parameter, full stop.

If an O(1) lookup ever matters more than O(profiles-in-this-space) trial application (it almost certainly won't at the scale of "how many distinct bank export shapes does one Space see" — likely single digits), the fast-path key to hash is the *sorted set of headers the mapping's values reference* — not the full header list — since that signature, unlike the full-header-list version, is unaffected by an unrelated column being added. But this is an optional micro-optimization, not a precondition; don't build it speculatively (see §10).

## 6. If a bank adds a new optional column next month, should the same mapping profile still match?

Yes, and this is the direct consequence of §5: with a subset/dependency-based match (trial-applying `applyExplicitMapping()`), an added column the mapping never referenced is invisible to the match — it's simply one more unmapped header, exactly as it would be for `detectColumns()` today. With Part B's originally-proposed full-header-set hash, the same scenario silently breaks the profile: the hash changes, the lookup misses, and the user is dropped back into "map this file manually" for a file that's 95% identical to the one they mapped last month. That failure mode is worse than having no saved-profile feature at all, because it fails exactly when the feature was supposed to prove its value (a recurring source the user has already taught the system once). This should be treated as a correctness requirement for 4D-5b's matching design, not a nice-to-have refinement.

The inverse case is correctly a non-match either way: if a bank *renames* a column the mapping depends on (not adds — renames or removes), `applyExplicitMapping()` correctly returns an error, the saved profile correctly fails to apply, and the user is correctly prompted to remap. That's the right failure mode for that case, distinct from the added-column case above.

## 7. Should mapping profiles belong to User, Space, Institution, Import source, or some combination?

Space-scoped, keyed by the mapping's own referenced-header set (§5) — affirming Part B's §4 conclusion, which is well-argued and doesn't need re-deriving: per-account is too narrow (the same bank produces the same shape across every account a user holds there), per-user doesn't let a teammate in the same Space reuse a mapping, and per-institution isn't usable as a lookup key because `institutionId` is frequently null on manually-uploaded files. Space-scoping also isn't an isolated choice here — it's consistent with how every other piece of this schema already scopes derived data (`FinancialAccount.ownerSpaceId`, `SpaceAccountLink`, `ImportBatch.financialAccountId` chaining up to a Space). A user-scoped `ImportMappingProfile` would be the actual outlier against the rest of the schema, not the natural default. `institutionLabel` and `source: ImportSource` as informational, non-key metadata (per the sequencing proposal's §6 refinement) are good additions for a future profile-list UI and shouldn't be part of the lookup key.

## 8. Is `NormalizedTransaction` the correct canonical contract going forward, or should it evolve?

Its required core is right and shouldn't change — it's now been proven, not just designed, across two genuinely different source shapes (CSV's string-keyed rows, Excel's typed cells), which is real evidence, not a guess. Its optional surface should evolve once, in 4D-5b, per the sequencing proposal's §5: `transactionType`, `balanceAfter`, `currency`, `rawMetadata`, all additive and optional, added when something is actually about to populate them rather than ahead of need. No disagreement with either prior doc here.

One gap neither prior doc raised: there's no per-row capture of what was actually read from the source for a failed or ambiguous row. `NormalizedTransaction.error` is a string; there's nothing alongside it holding the raw values that produced that error. That's fine today (errors surface in `errorSummary` as `{row, reason}` pairs, which is enough for the current "see a count of failures" UX) but worth keeping in mind before 4D-5c's preview screen is designed — a preview that can't show "here's what we read from your file for this row" alongside a flagged error is a weaker preview than one that can. Not a blocker for 4D-5b; flagging it now so it's a deliberate choice in 4D-5c's design rather than a retrofit.

## 9. Does anything in today's architecture make Step 5 harder than it needs to be?

Less than it could, and one place it's genuinely easier: per the sequencing proposal's §7, a `NormalizedTransaction`-shaped contract validated across three sources before Step 5 even starts means Step 5's import-adapter half is largely proven already. Two concrete friction points worth flagging as Step-5 homework rather than blind spots, since both are already implicitly visible in the code:

First, `route.ts`'s batch-create → per-row-classify → finalize block (the ~90 lines after column resolution) is inline in the route, not its own reusable function. That's correctly not extracted yet — there's no second caller — but it's the same shape of decision 4C already made once for the fingerprint helper (extract when a second concrete consumer exists, not before). QuickBooks (4D-4) will be that second consumer; flagging now so the extraction is recognized as due then, not treated as new scope discovered mid-4D-4.

Second, `SignConvention` (this module's file-upload-specific "what does a positive number mean" contract) and Plaid sync's own sign-flip convention in `lib/plaid/syncTransactions.ts` are two independent representations that happen to target the same outcome (creditPositive) but aren't unified through one shared type. Step 5's "shared normalized format every adapter maps into" will need to reconcile these explicitly — not hard, since they already agree on the target convention, but it's a real seam, not a hypothetical one.

Neither blocks 4D-5b. Both are worth a one-line mention in whatever document eventually scopes Step 5, so they're treated as already-known rather than rediscovered.

## 10. If you were the architect reviewing this fresh today, what would you raise?

Three things, in priority order:

The header-signature matching primitive (§5/§6) is the one finding in this review that's a correctness issue, not a style preference — it should change before `ImportMappingProfile`'s schema is finalized, because the schema shape (`mapping: Json` vs. a `headerSignature: String` hash column with separate comparison logic) follows directly from which matching approach is chosen, and reversing that decision after the table exists is a real migration, not a code change.

The two small abstractions in §3/§4 (shared validation, centralized resolution) aren't urgent in isolation, but 4D-5b is the moment to do them — the cost of doing them now is "touch `csv.ts` once, thoroughly" and the cost of deferring is "touch `route.ts` and `excel.ts` again for 4D-5b, then again for 4D-5c, each time hand-extending a ternary that should have been a single call." This project's own standing discipline (don't abstract before a second concrete need) argues for doing it now, specifically because 4D-5b *is* that second concrete need arriving — waiting further would be inconsistent with that discipline, not an application of it.

The `ImportBatch.resolvedColumnMapping` gap (§2) is the one piece of "missing audit trail" that's specific to this pipeline rather than a restatement of the project's already-known, already-tracked gap that CSV/Excel imports don't write an `AuditLog` entry at all (named explicitly in `D2_ROADMAP.md`'s own "4D (remainder)" row, still open, not part of this review's scope to re-litigate but worth noting it's still outstanding).

On over- vs. under-engineering, directly: this pipeline is not over-engineered. Every persistence/UI/fuzzy-matching deferral so far has had a stated, concrete reason tied to "nothing depends on this yet," not just caution for its own sake — and that's the same discipline the decision matrix applies elsewhere (D9's `SpaceTemplate`-alone scoping, D10's deferred-list ratification, D12's "add the gate when the feature needing it exists"). It is mildly under-engineered in exactly one place worth naming: the resolution-call-site duplication in §4, which existed the moment 4D-5a shipped a second resolver and will compound with a third. That's not a sign of sloppiness — `applyExplicitMapping()` was correctly scoped to "smallest safe slice" for 4D-5a alone — it's a sign that 4D-5b is precisely the right moment to pay down a debt that was reasonable to defer once and unreasonable to defer twice.

The roadmap itself doesn't need to change again. The matching mechanism inside 4D-5b's own checklist does, before that checklist is written.
