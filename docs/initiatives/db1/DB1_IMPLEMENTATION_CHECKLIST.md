# DB1 — Physical Schema Cleanup (Workspace → Space) — Implementation Checklist

**Status:** PLANNING ONLY — do not implement until approved.
**Prepared:** 2026-07-05. **Branch (planned):** `feature/v2.5-spaces-completion` (or a dedicated `feature/db1-schema-rename`).
**Governing entry:** STATUS.md §3 (DB-x track) / §5 (DB1 block).

Goal: make the physical PostgreSQL vocabulary match the application vocabulary (already 100% "Space"), by removing the 23 legacy Workspace-era `@map`/`@@map` annotations and renaming the underlying tables, columns, enum types, indexes, and constraints — **metadata-only**, zero data movement, zero application-code change.

**Absolute rule:** the migration is hand-authored with `ALTER … RENAME` only. Prisma's autogenerated `DROP`/`CREATE` diff is **discarded**. If the final migration file contains `DROP TABLE`, `CREATE TABLE`, `DROP COLUMN`, or `ADD COLUMN` for any of these objects, the migration is wrong.

## 1. Inventory re-confirmation (Q1)

23 mappings in `prisma/schema.prisma`, all legacy Workspace rename. Zero FinTracker / reserved-word / Prisma-limitation / compatibility-only maps.

**`@@map` — 6 tables:** `Space→Workspace`, `SpaceMember→WorkspaceMember`, `SpaceInvite→WorkspaceInvite`, `SpaceGoal→WorkspaceGoal`, `SpaceDashboardSection→WorkspaceDashboardSection`, `SpaceSnapshot→WorkspaceSnapshot`.

**`@@map` — 5 Postgres enum types:** `SpaceType→WorkspaceType`, `SpaceCategory→WorkspaceCategory`, `SpaceDashboardTab→WorkspaceDashboardTab`, `SpaceMemberRole→WorkspaceMemberRole`, `SpaceMemberStatus→WorkspaceMemberStatus`.

**`@map` — 12 columns:** `workspaceId` on `SpaceMember`, `SpaceInvite`, `AiAgent`, `Account`, `DuplicateAccountCandidate`, `SpaceGoal`, `SpaceDashboardSection`, `SpaceSnapshot`, `AiAdvice`, `AuditLog` (10); plus `User.preferredWorkspaceId` and `FinancialAccount.ownerWorkspaceId` (2).

Two tables reference the space with **already-clean** physical column/constraint names and need **no** change: `SpaceAccountLink.spaceId` (`SpaceAccountLink_spaceId_fkey`) and `ImportMappingProfile.spaceId` (`ImportMappingProfile_spaceId_fkey`). Their FKs point at `Workspace` by OID and auto-follow the table rename.

## Exact SQL plan

Ordering inside the single migration: (A) enum types → (B) tables → (C) columns → (D) primary-key constraints → (E) foreign-key constraints → (F) unique indexes (`_key`) → (G) plain indexes (`_idx`). All statements are DDL and run in one implicit transaction (Postgres DDL is transactional), so the migration is all-or-nothing. No target name pre-exists, so there are no collisions.

## Rollback plan (Q10)

Prisma has no down-migrations; three options, cheapest first for a pre-prod local DB:

**Option 1 — reset (preferred locally).** Revert the `schema.prisma` edit and delete the new migration folder (both uncommitted), then `npx prisma migrate reset` + reseed. Rebuilds the DB from history; no Workspace→Space migration in play. Zero-risk because there are no customers.

**Option 2 — restore dump.** Restore the `pg_dump` taken in checklist step 1.

**Option 3 — inverse migration.** If the migration was already committed/shared, apply an inverse migration that runs the 73 renames backwards in reverse section order (G→F→E→D→C→B→A). The inverse of each statement swaps the two identifiers, e.g.:

```sql
-- reverse (G) indexes, then (F) unique indexes, then (E) FKs, (D) PKs, (C) columns, (B) tables, (A) types
ALTER INDEX "Space_type_idx" RENAME TO "Workspace_type_idx";
-- … (all _idx) …
ALTER INDEX "SpaceMember_spaceId_userId_key" RENAME TO "WorkspaceMember_workspaceId_userId_key";
-- … (all _key) …
ALTER TABLE "SpaceMember" RENAME CONSTRAINT "SpaceMember_spaceId_fkey" TO "WorkspaceMember_workspaceId_fkey";
-- … (all _fkey) …
ALTER TABLE "Space" RENAME CONSTRAINT "Space_pkey" TO "Workspace_pkey";
-- … (all _pkey) …
ALTER TABLE "SpaceMember" RENAME COLUMN "spaceId" TO "workspaceId";
-- … (all columns) …
ALTER TABLE "Space" RENAME TO "Workspace";
-- … (all tables) …
ALTER TYPE "SpaceType" RENAME TO "WorkspaceType";
-- … (all types) …
```

The full inverse set is mechanical (swap left/right of every §A–§G statement). It also requires re-adding the `@map`/`@@map` lines to `schema.prisma`.

## Risk notes

- **R1 — Prisma autogenerated destruction (highest).** Letting `migrate dev` (without `--create-only`) apply its diff would `DROP`/`CREATE` tables = total data loss. Mitigation: `--create-only` + full manual SQL replacement + the step-5 grep gate + human review that the file is ALTER-only.
- **R2 — Missed index/constraint rename → persistent drift.** Because the tables are currently `@@map`'d, Prisma computes expected index/constraint names from the *mapped* (Workspace) name today; after removing the maps it expects the Space-convention names. Any object not renamed makes `migrate dev` propose a corrective migration. Mitigation: the 73-statement list is derived from the migration ledger; step-6 "no follow-up migration" is the definitive detector; the catalog query is the backstop.
- **R3 — Name-collision on rename.** None found — no `Space*` target name pre-exists (`SpaceAccountLink_*`/`ImportMappingProfile_*` don't overlap the targets). Re-verify with the catalog query if the schema changed since 2026-07-05.
- **R4 — Other environments.** Pre-prod: local + possibly a preview DB. The migration renames correctly wherever prior migrations were applied. It is **not** `IF EXISTS`-guarded, so it must run exactly once per environment (standard migration behavior).
- **R5 — Orthogonal legacy references.** `phase0-seam-gates.ts` / `space-account-link.ts` mention the retired `WorkspaceAccountShare`; intentionally out of scope. Do not fold WAS cleanup into this migration.
- **R6 — Effort/volume.** 73 hand-written statements (plus inverse). Low conceptual risk, but transcription risk — copy the exact identifiers from this doc / the ledger, don't retype.

**Overall risk: LOW** for a pre-production local DB — metadata-only DDL, no row movement, disposable data, and a definitive automated in-sync check. Contingent entirely on R1/R2 mitigations holding.
