# v2.0.1 — Backend & Data Foundation

**Status:** Release
**Release Date:** 17 June 2026
**Git Tag:** `v2.0.1`
**Scope:** the last release before Fourth Meridian / Spaces planning begins. v2.0.1 hardens the data model and sync pipeline that v2.1.0 — the Fourth Meridian identity, Spaces-first, and UI/UX implementation release — will sit on top of. No product-language or rebrand work is included here; this is foundation only.

---

## Release Philosophy

v2.0.1 is the backend/data maturity release. It stabilizes synchronization, reconciliation, snapshots, archive/trash lifecycles, and semantic consistency across the schema — the groundwork the rest of the platform builds on. It intentionally does not include the Fourth Meridian rebrand or a Spaces-first UX; those are deliberately out of scope here.

v2.1.0 is intended to become the first release of Fourth Meridian. It introduces the platform identity, Spaces-first information architecture, module-oriented navigation, and a complete mobile-first UI/UX redesign built on the backend foundation established in v2.0.1.

## 1. New Features

**Plaid linking.** Users connect an institution through Plaid Link; `POST /api/plaid/exchange-token` exchanges the `public_token` for an access token, encrypts it (AES-256-GCM, `lib/plaid/encryption.ts`), and upserts a `PlaidItem` keyed on Plaid's item id. Re-linking the same institution is treated as a token refresh, not a duplicate connection.

**Account import.** On exchange, every account Plaid returns is imported as a `FinancialAccount` + `AccountConnection` + `WorkspaceAccountShare` triple — not just a raw account row. Plaid-sourced metadata (`plaidName`, `officialName`, mask, type) is frozen at import time and never overwritten by later syncs, so a user rename survives refreshes. Manual (non-Plaid) account/asset import is a separate path — see "manual asset classification" below.

**Duplicate/reconnect reconciliation.** Plaid does not guarantee `account_id` stability across reconnects — the same real-world account can come back with a new id. `lib/accounts/reconcile.ts` handles this in two layers: an exact-identity match (`plaidAccountId`, or `ownerUserId`+`walletAddress` for crypto wallets), then a fingerprint fallback (institution/institutionId + mask + type + name fields) against archived accounts when no exact match exists. The system never hard-deletes a losing duplicate — it folds the duplicate's transactions, goal contributions, debt profile, and workspace shares into the canonical surviving row. This same logic is reused by account restore (`/api/accounts/[id]/restore`) so restoring an archived account that now collides with an active one merges silently instead of producing a visible duplicate.

**Account archive/restore.** `FinancialAccount` and `AccountConnection` use a soft-delete (`deletedAt`) pattern. Restore re-activates the account, its connections, and any revoked `WorkspaceAccountShare` rows — unless reconciliation (above) determines it should be merged into an existing active account instead.

**Workspace archive/trash lifecycle.** Added in v2.0.1 by mirroring the existing `WorkspaceGoal` archive/trash pattern onto `Workspace` itself: `archivedAt` (hidden from nav, fully restorable) and `deletedAt` (moved to trash, restorable until permanently purged). Migration is purely additive — every pre-existing workspace is implicitly active.

**Refresh pipeline.** `lib/plaid/refresh.ts` is the single reusable pipeline behind the manual Refresh action (and the intended call site for a future cron job or webhook handler). For each active `PlaidItem` it runs, in order: (1) balances/account metadata via `accountsGet`, matched only against existing accounts — refresh never creates or restores an account, that's relink's job; (2) investment holdings via `investmentsHoldingsGet`, best-effort and non-fatal; (3) transactions via the shared incremental sync function; (4) `WorkspaceSnapshot` regeneration for every workspace the refreshed accounts are shared into. Every step is idempotent and safe to re-run.

**Transaction import.** `lib/plaid/syncTransactions.ts` wraps Plaid's cursor-based `/transactions/sync` so initial import, manual sync, the (currently dormant) scheduled job, and a future webhook handler all share one implementation. It resumes from a persisted cursor, loops on `has_more`, upserts on `plaidTransactionId` (safe against retries/duplicate webhook delivery), and flips Plaid's debit/credit sign convention to match the convention used internally by the financial module of Fourth Meridian (positive = money in).

**Snapshot generation.** Before v2.0.1, the only code path that ever wrote a `WorkspaceSnapshot` row was the seed script — real workspaces had zero snapshot history regardless of how fresh their balances were, so the Cash History / Net Worth charts rendered blank. `lib/snapshots/regenerate.ts` adds a production writer: `regenerateWorkspaceSnapshot` upserts today's row from live `FinancialAccount` balances (idempotent on `[workspaceId, date]`), and `regenerateSnapshotsForAccounts` fans that out to every workspace sharing a given set of accounts. Wired into both the Refresh pipeline and the initial Plaid Link import.

**Chart first-day placeholders.** `ChartFirstDayPlaceholder.tsx` renders a real headline value plus "Started tracking today" / "Check back tomorrow to begin seeing trends" copy when a workspace has exactly one `WorkspaceSnapshot` row, avoiding a broken-looking trend line on day one.

**cashOnHand / actionReady terminology cleanup.** Genuine column renames (not new fields) retiring legacy "play cash" language: `WorkspaceSnapshot.cashToPlay` → `cashOnHand`, `AiAdvice.playReady` → `actionReady`. Done in the final migration of the release.

**Manual asset classification — implemented.** Manual entries (`POST /api/accounts/manual`) are created as `FinancialAccount` rows with `type=other` and `syncStatus=manual`, and accept an `assetKind` field (`real_estate` / `vehicle` / `equipment` / `other`) recorded in the audit log at creation time. `lib/account-classifier.ts` is the single source of truth that buckets every account — manual or linked — into liquid / investments / digital assets / real assets / liabilities, used by both the dashboard and snapshot regeneration. `assetKind` is descriptive metadata layered on top of this bucketing, not yet a distinct classification dimension on its own.

## 2. Schema Changes

**New tables:** `FinancialAccount`, `AccountConnection`, `WorkspaceAccountShare`, `DuplicateAccountCandidate`, `WorkspaceGoal`, `GoalContribution`, `GoalCheckIn`, `WorkspaceInvite`, `WorkspaceDashboardSection`, `DebtProfile`, `RecoveryCode`, `UserSession`, `PlatformSetting`.

**New enums:** `UserRole`, `EmploymentStatus`, `UseCase`, `InviteStatus`, `WorkspaceCategory`, `WorkspaceMemberStatus`, `ShareStatus`, `DuplicateStatus`, `AccountOwnerType`, `GoalCategory`, `GoalStatus`, `WorkspaceDashboardTab`, `GoalType`. `VisibilityLevel` gained `BALANCE_ONLY`/`FULL`; `WorkspaceCategory` gained `RETIREMENT`/`DEBT_PAYOFF`/`EMERGENCY_FUND`/`CUSTOM`; `PlaidItemStatus` gained `REVOKED`.

**Notable new columns:** `Workspace.description`, `Workspace.isPublic`, `Workspace.category`, `Workspace.archivedAt`/`deletedAt`; `WorkspaceMember.status`/`revokedAt`/`revokedById`; `FinancialAccount.interestRate`/`minimumPayment`/`debtSubtype`/`plaidName`/`officialName`/`displayName`; `WorkspaceGoal.goalType`/`habitFrequency`/`currentStreak`/`longestStreak`/`lastCheckIn`/`spendingCategory`/`linkedAccountId`/`targetReductionAmount`/`targetReductionPct`/`snapshotBalance`/`archivedAt`/`deletedAt`; `Transaction.financialAccountId` (new, alongside the now-nullable legacy `accountId`); `User.role`, profile fields (name, employment, use case, encrypted DOB, username), `preferredWorkspaceId`, `forcePasswordReset`, `lastBriefViewedAt`; `AuditLog.userAgent`/`performedByAdminId`.

**True column renames** (existing column renamed in place, not a new column added):
- `PlaidItem.plaidItemId` → `externalItemId` — done ahead of future provider abstraction, so the column name no longer implies Plaid specifically.
- `WorkspaceSnapshot.cashToPlay` → `cashOnHand` — retires legacy "play cash" language; the UI card was already titled "Cash on Hand."
- `AiAdvice.playReady` → `actionReady` — same "play" metaphor family, same reason.

Worth noting separately: the schema's `WorkspaceSnapshot` model carries a header comment that it was "renamed from `DailySnapshot`," but that rename predates the v2.0.1 migration history (it was already `WorkspaceSnapshot` at the initial migration), so it's mentioned here for completeness rather than listed as a migration in section 3.

## 4. Known Limitations

- **Daily cron not implemented.** `jobs/scheduler.ts` defines `startScheduler()` (sync-banks every 4h, sync-crypto stub hourly, take-snapshot daily, run-ai-advice, purge-trash daily) but it is never invoked anywhere — there's no `instrumentation.ts` hook wiring it up. The jobs are written but dormant.
- **Plaid webhooks not implemented.** No webhook endpoint exists yet. `syncTransactionsForItem` is explicitly written to be webhook-ready (`SYNC_UPDATES_AVAILABLE` would call it directly), but the endpoint itself — signature verification, retries, public exposure — is out of scope for this release.
- **Historical snapshot backfill not implemented.** `regenerateWorkspaceSnapshot` only ever upserts *today's* row. There is no function that reconstructs past dates from historical balance/transaction data — a workspace's snapshot history only starts accumulating from whenever this release first runs for it.
- **Crypto/xpub tracking not implemented.** Wallets are tracked by manually-entered address only; there is no xpub/zpub/ypub-based derivation or automatic on-chain balance sync. `jobs/sync-crypto.ts` is a stub (`export {}`).
- **Archive transaction viewer not implemented.** There is no UI/API surface for viewing transactions that belonged to an archived or merged-away account.
- **Provider abstraction deferred.** `PlaidItem` and Plaid-specific client/encryption code remain Plaid-specific by design. The `externalItemId` rename (migration 20) is a deliberate first step toward this, but no actual abstraction layer exists yet.
- **Workspace-to-Space code rename deferred.** Per project constraints, `Workspace` stays the model/code name through this release; "Space" exists only as a planning concept, not in schema or code.
